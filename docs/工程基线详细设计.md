# 基于 TPM 2.0/vTPM 的密钥管理与数据加解密系统工程基线详细设计

统一密钥层次为 `TPM/vTPM -> NRWK/TPM object -> CRK -> DEK -> Data`。`crk_aad_digest` 是应用层完整性摘要；原生 provider 的 TPM policy/对象授权才是硬件强制边界。

版本：v1.3
状态：与需求规格 v1.3、总体设计 v1.3 对齐
适用范围：工程基线

## 1. 基线目标

工程基线是当前唯一交付边界，不再拆分。基线必须同时覆盖密钥生命周期、受控导入、服务端 Encrypt/Decrypt、批量 Encrypt/Decrypt、租户信封配置、认证授权、审计链、生命周期任务、签名策略治理和基础容量保护。基线还必须覆盖 Key 显式 ID、Key 有效期、过期 Key 延期、按格式 Envelope 字段配置、审计清理、worker 参数可观测和 outbox 大列表可用性。

基线的核心安全承诺是：明文密钥材料不得离开服务端/key-resolver/TPM 可信边界；任何 API、日志、审计、数据库、备份或 UI 响应不得返回 DataKey、DEK、CRK 或等价明文密钥材料。

## 2. 范围与边界

| 交付主题 | 设计内容 | 边界约束 |
| --- | --- | --- |
| Key CRUD 与生命周期 | 创建、查询、列表、元数据更新、计划销毁、取消销毁、启用、禁用、轮转、有效期设置和延期。 | 删除语义为计划销毁；状态迁移由领域层控制；不得物理删除审计证据或历史版本元数据；到期 Key 加解密 fail-closed。 |
| 受控导入 | 外部密钥材料进入密钥面校验并封装。 | 只允许导入，不允许导出；导入材料不落库、不进日志、不回显。 |
| 服务端加解密 | Envelope v1、数据面 opaque AAD、系统内部 AAD canonical、AES-256-GCM、SM4-GCM、AES-256-ECB 与 SM4-ECB。 | 同步对象上限 64 KiB；工程基线仅支持 `encrypt_decrypt` 用途；同一 Key 绑定唯一 suite，GCM 与 ECB 密钥不得跨模式复用。ECB 忽略 caller AAD，仅用于兼容性场景，不提供 GCM 等价的完整性或 AAD 认证能力。 |
| 批量 API | 批量 Encrypt/Decrypt 最多 100 条，均采用 `entries` 输入模型。 | 共享认证和外层限流；每条独立校验 Key、状态、权限、AAD bytes 和策略；GCM 条目额外校验 nonce；解密成功只返回 base64 明文。 |
| 策略治理 | 签名策略包、版本化、热加载、失败保留最后有效策略。 | 未签名、验签失败或语义降级的策略不得生效。 |
| 审计链与生命周期任务 | 追加式审计链、验证接口、按授权删除审计事件、outbox、worker 幂等任务和 worker 参数查询。 | 审计载荷只写白名单字段；worker 重试不得重复销毁或造成状态错乱；删除审计事件仅用于授权清理场景并应重置对应链头。 |
| 租户配置 | 信封格式默认值、允许列表和按格式 profile 查询/更新。 | 租户身份来自认证上下文，路径或请求体仅做一致性校验；外部格式仅保留 `json-v1` 与 `configurable-json-v1`。 |

| --- | --- | --- |
| `management-api` | Key CRUD、生命周期状态操作、Key 有效期延期、租户配置和策略治理路由。 | 执行数据面加解密；返回 CRK/DEK 明文；绕过状态机。 |
| `crypto-api` | Encrypt/Decrypt、批量加解密、Envelope 编解码、`aad_b64` 数据面认证输入、租户 Envelope 格式选择。 | 改变 Key 生命周期；访问 TPM/vTPM；向客户端返回密钥材料。 |

| `domain` | Key/KeyVersion 状态机、租户 ABAC、策略校验、错误码映射。 | 处理 HTTP 细节或持久化密钥明文。 |
| `key-resolver` | CRK 解封、DEK wrap/unwrap、短 TTL 内部 DEK lease。 | 暴露公网 API；承担业务授权决策；记录明文密钥。 |
| `tpm-provider` | TPM/vTPM provider 抽象、生产级 TSS/ESAPI 目标实现、受限 `tpm2-tools` fallback、软件测试 provider。 | 数据面直连；依赖 shell；记录 CRK 明文；把命令行 provider 标记为生产首选。 |

| `audit` / `auditchain` | 结构化审计、字段白名单、哈希链和验证。 | 记录明文、token、DEK、CRK、完整 Envelope 或 wrapped material。 |
| `lifecycle-worker` | 领取并幂等执行销毁到期、Key 过期扫描、缓存失效、审计转发等任务，并暴露脱敏运行参数。 | 绕过领域状态机直接修改最终状态。 |

## 4.1 Ops Plane 设计

行业参考给出的共同模式是：高危运维必须有延迟销毁、可取消/可恢复窗口、审计不可绕过、专用权限和受控恢复路径。例如 AWS KMS 的 key deletion 使用 waiting period；Google Cloud KMS 的 key version destruction 也采用 scheduled destruction 和 restore window；Azure Key Vault soft-delete/purge protection 将删除和最终 purge 分离；HashiCorp Vault 强调 audit devices，并将 seal/unseal、rekey、rotate 等操作作为 operator 级能力。工程基线吸收这些模式，但不直接复制云厂商产品语义。

Ops Plane 采用“白名单动作 + 审计预写 + 脱敏响应”的模型：

| 能力域 | 动作 | 最小 scope | 数据边界 |
| --- | --- | --- | --- |
| 只读诊断 | 查询 TPM/provider、resolver、repository、policy、audit、worker、outbox 状态 | `ops:read` | 仅返回状态、hash、计数、时间戳和错误码。 |
| CRK 运维 | 查看 CRK envelope 脱敏状态、修复 `crk_aad_digest`、刷新 resolver | `ops:read` / `ops:repair` | 不返回 `wrapped_crk` 原文；只返回长度、digest、版本、node/cluster 元数据。 |
| 生命周期运维 | 重试失败 job、取消卡死 job、查看销毁倒计时 | `ops:repair` / `ops:breakglass` | 不允许绕过 24 小时 Key 销毁冷静期；销毁 job 执行前必须复核 Key 仍为 `DESTROY_PENDING`。 |
| Outbox 运维 | 查看积压、重放失败事件、标记不可恢复事件 | `ops:repair` / `ops:breakglass` | 不修改审计链历史；重放必须幂等。 |
| 数据库运维 | 查看迁移版本、容量等级、连接健康、锁/积压摘要 | `ops:read` | 不提供任意 SQL；不导出表数据；不返回敏感列。 |
| 应急治理 | break-glass 修复、隔离异常节点、强制释放过期 lease | `ops:breakglass` | 请求必须携带 `reason`、`ticket_id`、`Idempotency-Key`；必须记录 preflight、执行结果和影响范围。 |

所有变更型 Ops 动作必须满足：

1. 先执行 dry-run/preflight，确认动作对象、当前状态、影响范围和可回滚性。
2. 审计链预写 `ops.<resource>.<action>.requested`，审计失败时 fail-closed。
3. 通过 repository/service 层方法执行，不允许拼接 SQL 或绕过领域状态机。
4. 响应只包含动作 ID、对象 ID hash、状态、摘要和下一步建议。
5. 结束后写入 `ops.<resource>.<action>.succeeded` 或 `.failed`，失败事件不得包含敏感原文。

### 4.1.1 包结构与装配

Ops Plane 由独立 `internal/api/ops` 包实现，通过 `server.Deps.OpsHandler` 注入，不与管理面或数据面 handler 复用授权入口。`baselineapi` 只保留 audit、policy、lifecycle 查询、batch crypto 和 system status。变更动作只调用 repository/service 白名单方法：`RetryLifecycleJob`、`ReplayOutboxEvent`、`UpdateCRKNodeEnvelope` 与 resolver 的受控缓存替换；不存在任意 SQL、shell、脚本、备份或密钥材料导出入口。

Repository 的运维能力包括：

- `RetryLifecycleJob`：仅把合法的失败任务重置为 `PENDING`，状态判断和并发约束由 repository 保证。
- `ReplayOutboxEvent`：按幂等语义把指定事件恢复为可处理状态。
- `DatabaseDiagnostics`：返回类型化、脱敏的运行态快照；PostgreSQL 使用连接池统计、系统统计视图和固定聚合查询，避免任意查询和业务表全量读取。表统计明确区分近似行数、表字节和索引字节。

### 4.1.2 统一动作治理

普通 repair 与 break-glass 使用同一动作模板，但拥有不同 URL 前缀和 scope，不允许 `ops:breakglass` token 通过普通 repair 路径隐式降级：

1. 严格 JSON 解码动作请求；必填 `reason`、`ticket_id`，break-glass 额外必填 `impact_scope` 且 `operator_confirmation=true`。
2. 校验 `Idempotency-Key`，并绑定 principal、method、path 和 request hash；同键异请求返回 `IDEMPOTENCY_CONFLICT`。
3. 生成 preflight：目标、当前状态、影响范围、可回滚性、预计结果和 flow。
4. 预写 `.requested` 审计，字段含 ticket、reason、幂等键 hash 和 preflight；失败即 503 且不执行动作。
5. 预留幂等记录；冲突时补写 `.aborted`，保持 requested → final 审计配对。
6. 通过 repository/service 执行动作；成功写 `.succeeded`，失败写 `.failed` 和稳定错误码。最终审计不可用时隐藏动作结果并返回 503。
7. 响应只返回状态、ticket、preflight 和对象脱敏摘要，不返回 CRK/DEK、wrapped material、payload 原文、SQL 结果或堆栈。

break-glass 只是更强审计的白名单应急通道，不提供 lifecycle cancel、节点隔离、强制释放有效 lease 或直接 purge。任何 Key material 最终销毁仍必须经过 `DESTROY_PENDING` 与 24 小时冷静期。

### 4.1.3 健康与数据库诊断

`GET /ops/health` 聚合 TPM provider、resolver、repository、worker、audit chain 和 policy manager 六个维度，每个维度返回 `ok`、`warn` 或 `degraded` 及脱敏摘要。software、swtpm、tpm2-tools 和 memory repository 只产生工程 fallback 警告；生产配置在启动阶段直接拒绝这些边界。

`GET /ops/db/status` 返回 driver、`ok|warn|degraded`、观测时间、连接延迟、连接池、主备角色、迁移版本、数据库/表/索引容量、会话/锁/长事务、复制摘要、lifecycle/outbox 积压、全局 Key 聚合、CRK envelope digest 与领域一致性检查。领域一致性固定检查孤儿 KeyVersion、`DESTROYED` KeyVersion 材料是否清空、过期但仍活跃的 DEK/nonce lease。错误只使用稳定原因码；原始 PostgreSQL 错误、DSN、SQL、记录值、Key ID、标签、wrapped material 和完整 Envelope 均不得出现在响应中。

数据库诊断状态由后端统一判定，前端不得自行把不可用字段解释为健康：连接延迟超过 100 ms 为 `warn`、超过 500 ms 为 `degraded`；连接池使用率达到 80%、存在锁等待或超过 30 秒事务为 `warn`；迁移版本不一致、孤儿 KeyVersion 或销毁后仍保留材料为 `degraded`；过期活跃 lease 为 `warn`。受数据库角色权限或 backend 能力限制的探针进入 `unavailable`。备份由外部备份系统负责时返回 `managed_externally`，本系统不虚构无法独立验证的最近成功时间。

历史 `DESTROY_PENDING` Key 的最终销毁依赖 `destroy_due` lifecycle job；状态字段本身不会自动触发数据库层删除。启动阶段会执行一次 `ReconcileDestroyDueJobs`，为缺少 `PENDING/RUNNING destroy_due` job 的历史 `DESTROY_PENDING` Key 补建任务，补建任务的 `next_run_at` 使用 `keys.updated_at + 24h`。已失败的 `destroy_due` job 不自动绕过，必须通过 Ops retry 处理，以保留失败原因和审计链。

### 4.1.4 前端运维工作台

Dashboard 的轻量 Health 保持 5 秒查询周期，数据库重诊断降为 30 秒，只展示状态、连通性与延迟并提供 Database Operations 入口。独立 Database Operations 页面以 60 秒周期展示 Runtime、Data Protection、Domain Integrity、Diagnostic Coverage 和 Data Atlas；Data Atlas 只消费 Ops 全局聚合，不调用管理面 `/keys`。默认导航仅提供 Dashboard、Key Management 与 Encrypt/Decrypt；Database、Policy、Envelope、Batch、Audit 和 Lifecycle 位于用户显式开启的 Advanced controls。组件统一通过 `apiPaths` 构造路径。所有变更按钮在提交前收集 reason/ticket 并生成新的幂等键；成功后刷新相关 query。failed jobs、pending outbox、invalid CRK digest 和 audit degraded 可直接定位到对应面板。长 ID/hash 使用滚动或断行容器，错误、空状态和窄屏不得重叠。

PostgreSQL 启动迁移使用 `schema_migrations` 账本、事务和 transaction-scoped advisory lock：每个版本仅执行一次，成功后记录版本与名称。迁移失败将整体回滚并阻止服务启动；不得在每次启动时重复执行 schema DDL。启动期还应输出不含 DSN、token 或密钥材料的部署体检结论，覆盖运行环境、存储持久性、TPM provider 与 plane 隔离等级。

### 4.1.5 验收矩阵

Ops 验收至少覆盖：

- 普通 repair 与 break-glass 的 plane/scope 路由矩阵；管理面、数据面和缺 scope token 全部拒绝。
- 缺 reason、ticket、幂等键、break-glass impact/confirmation 时拒绝。
- 审计预写失败时业务状态不变；幂等预留冲突形成 `.aborted` 最终审计；最终审计失败时结果 fail-closed。
- retry 仅作用于允许状态；outbox replay 幂等；CRK digest unchanged/repair 均形成完整审计配对。
- 响应、日志、审计 metadata 和 UI 不含 token、明文、DEK、CRK、完整 Envelope 或 wrapped material。
- 健康卡的 ok/warn/degraded、空积压、错误状态、大列表、长 ID 和窄屏布局。
- 数据库诊断的迁移匹配/不匹配、延迟阈值、连接池使用率、锁等待、长事务、领域一致性异常和指标不可观测语义；响应敏感信息扫描必须覆盖 DSN、SQL、wrapped material 与记录值。

### 4.1.6 明确不提供的能力

任意 SQL、shell/脚本、备份导出、密钥材料导出、审计历史修改、绕过销毁冷静期、取消已进入销毁临界区的任务均不属于系统能力。节点隔离、过期 lease 强制释放和独立 metrics/Prometheus 采集若未来引入，必须另行设计白名单状态机、审计字段和负向测试，不能扩展为通用执行器。

## 5. 数据设计

基线至少包含以下逻辑对象：

1. `tenants`
2. `keys`
3. `key_versions`
4. `crk_versions`
5. `root_key_envelopes` / `crk_node_envelopes`
6. `dek_leases`
7. `nonce_usage_records` / `nonce_leases`
8. `crypto_policies`
9. `tenant_envelope_configs`
10. `audit_events`
11. `audit_chain_heads`
12. `outbox_events`
13. `lifecycle_jobs`
14. `idempotency_keys`

`keys` 必须包含可空 `expires_at`。对外 DTO 应包含有效期并在 `expires_at <= now` 且领域状态仍为 `ACTIVE` 时派生展示状态 `EXPIRED`。`tenant_envelope_configs` 必须包含 `default_format`、`allowed_formats` 和 `profiles`。每个 profile 至少包含 `format_id`、`adapter`，可包含 `field_mappings` 与 `extensions`；字段映射使用 `path`、`source`、`required`、`encoding` 和 `default_value` 描述外部格式到 CoreEnvelope、derived 值或 extension 值的关系。

关键约束：

1. `key_versions.wrapped_material` 只能保存受 CRK 或根材料保护的密文。
2. Key 创建、轮转、计划销毁、策略 reload 和审计链写入必须具备事务边界或等效原子性。
3. Repository 只持久化领域层已校验的状态，不决定状态迁移。
4. 内存仓库是工程基线验证实现，不宣称生产持久化能力；生产落地必须替换为 PostgreSQL repository、迁移和角色隔离。
5. JSON/profile Envelope 可以持久化和返回扩展字段；解密解析必须忽略不参与认证的扩展字段，并以核心字段重建 CoreEnvelope。GCM suite 按 canonical binary protected header 参与 AEAD 校验；ECB suite 忽略 caller AAD，不校验 `aad_hash`。
6. 外部 Envelope 格式仅保留 `json-v1` 与 `configurable-json-v1`。服务端内部 canonical 编码用于 GCM 认证边界和统一解析，不作为对外 API 格式暴露；租户 profile 只能改变 JSON 呈现，不能替代核心认证字段。
7. 数据面 caller AAD 是调用方提供的 opaque bytes，通过 `aad_b64` 进入 Encrypt/Decrypt。业务对象到 AAD bytes 的 canonical 步骤由调用方或 SDK 完成；服务端不得解析、重排或规范化，只做 base64 解码、最大 64 KiB 长度限制、`SHA-256(aad_bytes)` 审计和一致性校验；GCM suite 额外将其作为 AEAD AAD 使用。加密端和解密端的 AAD bytes 不一致时必须解密失败。
8. GCM 与 ECB 通过独立 suite_id 和 Key 绑定隔离：`AES_256_GCM`/`SM4_GCM` Key 只能解密同 suite Envelope，`AES_256_ECB`/`SM4_ECB` Key 只能解密同 suite Envelope。ECB JSON/profile 不携带 `nonce`、`tag`、`aad_hash`；密文采用 PKCS#7 padding；服务端不校验 ECB caller AAD。
9. 为降低接入复杂度，数据面支持最小加解密 body：Encrypt 只传 `plaintext` 时默认使用租户默认 `AES_256_ECB` Key（`t-default` 为 `default-ecb`，其他租户使用按 tenant 派生的默认 Key ID），首次使用时由服务端创建并封装 DEK；Decrypt 直接传 Envelope JSON，由服务端解析 Key、版本和 suite。系统不支持仅凭 raw ciphertext 推断默认 KeyVersion。
8. 生产级 TPM provider 使用进程内 go-tpm TSS 命令实现，`native`/`tss`/`esapi` 均装配该 provider；不 shell out、不依赖 PATH、不写 CRK 明文临时文件。配置的 PCR 集通过 `PolicyPCR` 约束，cluster/node/plane/baseline/policy canonical context 参与 TPM object authorization；错误 transport、policy、上下文或损坏 envelope 必须 fail-closed。`hardware`/`tpm2-tools` 仅为受控工程 fallback。
9. 数据库运维接口必须归属独立运维面 Ops Plane，并采用白名单动作模型，不得提供任意 SQL。CRK、DEK、wrapped material 等高危对象只允许脱敏元数据查看；任何会改变密钥面数据库状态的操作必须先写入审计链，审计失败必须 fail-closed。基线静态 token 必须区分管理、数据和运维三类 plane；运维操作使用 `ops-token-baseline`、`ops` plane 与 `ops:read`/`ops:repair` scope；应急操作必须额外要求 `ops:breakglass`。
10. Lifecycle worker 参数必须来自配置并可查询；默认值只作为工程基线兜底，不得硬编码为唯一行为。

## 6. 状态机

Key 状态：

| 状态 | 允许行为 |
| --- | --- |
| `ACTIVE` | 查询、更新元数据、加密、解密、轮转、禁用、计划销毁。 |
| `EXPIRED` | 派生展示状态；查询、更新元数据和延期；不得加密或解密；底层持久状态可保持 `ACTIVE` 以保留恢复路径。 |
| `DISABLED` | 查询、更新元数据、启用、计划销毁；不得新加密。 |
| `DESTROY_PENDING` | 查询、更新元数据、取消销毁、24 小时冷静期到期销毁；不得新加密；历史解密按策略控制。 |
| `DESTROYED` | 仅保留必要元数据和审计证据；不得启用、加密、解密、轮转或导出；可归档 tombstone 以从默认库存视图隐藏。 |

KeyVersion 状态：

| 状态 | 允许行为 |
| --- | --- |
| `ACTIVE` | 新加密和解密。 |
| `DECRYPT_ONLY` | 历史密文解密，不得新加密。 |
| `DISABLED` | 不得加密或解密，恢复必须审计。 |
| `DESTROYED` | wrapped material 不可恢复，不得加密或解密。 |

## 7. 验收门禁

工程基线发布前必须完成：

1. Key CRUD、生命周期状态机、受控导入、轮转、计划销毁和取消销毁测试通过。
2. Key 显式 ID、有效期展示、过期拒绝加解密、过期扫描和延期测试通过。
3. Envelope v1、数据面 opaque AAD、系统内部 AAD canonical、错 AAD、错版本、禁用 Key、过期 Key、销毁中 Key、GCM nonce 异常和批量部分失败测试通过。
4. Envelope 配置的默认格式、允许格式、profile 字段映射、JSON/profile 扩展字段解析、服务端 inspect/convert 和前端只读 inspector 测试通过。
5. 管理面、数据面、密钥面和审计面隔离可验证。
6. 签名策略包 reload、失败保留最后有效策略、审计链验证、授权审计删除和 worker 幂等测试通过。
7. 日志、审计、错误、指标和 UI 响应扫描不含 token、明文、DEK、CRK、完整 Envelope 或 wrapped material。
8. 内存仓库限制在 README 和设计中明确；任何生产部署声明必须先补齐 durable repository、迁移、备份恢复和数据库角色隔离。
