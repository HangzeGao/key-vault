# 基于 TPM 2.0/vTPM 的密钥管理与数据加解密系统需求规格说明书

版本：v1.3
适用范围：工程基线

## 1. 文档目的

本文定义工程基线的系统目标、功能需求、非功能需求、数据与 API 边界、发布门禁和需求变更准入规则。工程基线是当前唯一交付边界，不再拆分。

## 2. 术语与英文缩写

| 缩写/术语 | 英文全称 | 释义 |
| --- | --- | --- |
| TPM | Trusted Platform Module | 可信平台模块。本系统以 TPM 2.0 作为硬件可信根。 |
| vTPM | Virtual Trusted Platform Module | 虚拟可信平台模块。 |
| Key | Key | 租户维度下的业务逻辑密钥对象。 |
| KeyVersion | Key Version | 密钥版本。一次加密必须绑定到确定的 KeyVersion。 |
| CRK | Cluster Root Key | 集群/系统根密钥，用于保护下级密钥材料。 |
| DEK | Data Encryption Key | 数据加密密钥，应以受保护形态持久化。 |
| Envelope | Envelope Format | 密文信封格式，封装解密所需元数据。 |
| AAD | Additional Authenticated Data | 附加认证数据，参与完整性校验但不被加密。 |
| AEAD | Authenticated Encryption with Associated Data | 带附加认证数据的认证加密机制。 |
| nonce | Number Used Once | 单次使用值，GCM 下同一 KeyVersion 内不得复用。 |
| WAL | Write-Ahead Log | 高风险操作或审计事件的预写记录。 |
| ABAC | Attribute-Based Access Control | 基于属性的访问控制。 |
| SLO | Service Level Objective | 服务等级目标。 |
| EXPIRED | Expired Key | Key 有效期已到达后的派生状态。 |

## 3. 产品目标与边界

系统应建设为以 TPM 2.0/vTPM 为可信根的集中式密钥托管与服务端密码运算系统，面向内部业务系统和租户应用提供统一的密钥生命周期管理、受控密钥导入、服务端加解密、审计追踪和工程基线治理能力。

核心目标：

1. 根密钥和数据加密密钥明文只允许存在于服务端/key-resolver/TPM 可信边界内，禁止通过 API、日志、数据库、备份或普通文件离开该边界。
2. 以租户和业务 Key 为管理单元，提供创建、受控导入、查询、列表、元数据更新、启停、轮转、计划销毁、取消销毁、策略绑定和审计能力。
3. 通过服务端 Encrypt/Decrypt API 和批量 Encrypt/Decrypt API 对接业务系统，支持按 Key 绑定的算法、模式、用途和策略版本。
4. 固化 Envelope v1、数据面 opaque AAD、系统内部 AAD canonical、nonce 唯一性、错误码和 KeyVersion 状态机等长期契约。
5. 隔离管理面、数据面、密钥面和审计面，降低密钥材料暴露、越权操作、审计缺失和策略绕过风险。
6. 在工程基线中提供审计链、异步生命周期任务、签名策略治理、租户级配置、基础限流和容量保护。
7. 支持显式业务 Key ID、Key 有效期、过期拒绝加解密和授权延期，确保 Key 生命周期在查询、数据面和生命周期任务中表现一致。
8. 支持租户级 Envelope 格式和字段配置，在不破坏 Envelope v1 核心契约的前提下满足第三方格式适配和调试需求。

工程基线当前使用内存仓库支撑自动化验证和本地运行，不宣称生产持久化能力。任何生产部署声明必须先补齐 PostgreSQL repository、迁移、数据库角色隔离、备份恢复和恢复演练证据。

## 4. 安全不变量

| 编号 | 需求 |
| --- | --- |
| INV-01 | CRK 明文不得通过 API、数据库、日志、备份或普通文件导出。 |
| INV-02 | 数据库不得存储明文 DEK/CRK；DEK 入库前必须由 CRK 或受控根材料封装。 |
| INV-03 | 数据面不得直接访问 TPM/vTPM 或 CRK envelope。 |
| INV-04 | GCM nonce 不得复用；无法证明 GCM nonce 安全时必须拒绝对应 GCM 新加密；ECB suite 不使用 nonce。 |
| INV-05 | GCM 解密必须使用加密时完全一致的 caller AAD bytes；AAD hash 不能替代认证。ECB 忽略 caller AAD。 |
| INV-06 | Envelope、数据面 AAD bytes、系统内部 AAD canonical、错误码和状态机必须明确版本化演进。 |
| INV-07 | 租户身份必须来自认证上下文；请求体 tenant_id 仅可作一致性校验。 |
| INV-08 | 管理与数据权限必须使用不同 scope/token；数据 token 不得调用管理 API。 |
| INV-09 | 敏感字段不得写入日志，包括 token、明文、DEK、CRK、完整 Envelope、wrapped_key。 |
| INV-10 | 安全关键失败默认 fail-closed，不得降级为软件根密钥、跳过审计或重用 nonce。 |
| INV-11 | 未列入工程基线的能力必须显式关闭或返回稳定错误。 |
| INV-12 | 密钥只允许受控导入，不允许明文导出。 |
| INV-13 | 每个 Key 必须绑定唯一算法策略；不同算法、模式或用途不得复用同一逻辑 Key。 |
| INV-14 | Key 到期后必须 fail-closed；不得继续加密或解密，延期必须通过授权管理操作完成并审计。 |
| INV-15 | Envelope 字段扩展不得替代或弱化核心认证字段；JSON 扩展字段可被解析忽略，二进制格式扩展必须通过新版本格式演进。 |
| INV-16 | 数据面 caller AAD 是调用方提供的 opaque bytes；业务 canonical 由调用方或 SDK 完成，服务端不得解析、重排或规范化。 |

## 5. 功能需求

| 编号 | 需求 | 验收标准 |
| --- | --- | --- |
| EB-FR-01 | 系统应提供 Key 创建、受控导入、查询、列表、元数据更新、启用、禁用、轮转、计划销毁和取消销毁能力。 | API 可用；状态转换由领域层控制；删除请求采用计划销毁语义；响应不得包含密钥材料。 |
| EB-FR-02 | 系统应提供 Encrypt、Decrypt 能力。 | 加密返回 Envelope JSON；解密请求体直接使用 Envelope JSON 本体，校验 Envelope、AAD、租户、KeyVersion、权限后返回明文；响应不含明文密钥材料。Encrypt 未提供 `key_id` 时默认使用/自动创建租户默认 `AES_256_ECB` Key；最小加密 body 为 `{"plaintext":"<base64>"}`；最小解密 body 为加密响应返回的 Envelope JSON。 |
| EB-FR-03 | 系统应提供批量 Encrypt/Decrypt API。 | 最多 100 条；批量加密和批量解密均采用 entries 输入模型；共享认证和外层限流；每条独立校验 AAD、状态、权限和策略；GCM 条目额外校验 nonce；部分失败按条目返回稳定错误码。 |
| EB-FR-04 | 系统应支持 AES-256-GCM、SM4-GCM、AES-256-ECB 和 SM4-ECB。 | 每个 Key 绑定唯一 suite；GCM 与 ECB 密钥不得跨模式复用。ECB 忽略 caller AAD，JSON/profile Envelope 不携带 `nonce`、`tag`、`aad_hash`，仅用于兼容性场景，不提供 GCM 等价的完整性或 AAD 认证能力。 |
| EB-FR-05 | 系统应实现 Envelope v1、数据面 opaque AAD 和系统内部 AAD canonical。 | 固化测试向量；未知 version、flags、suite_id 或长度异常必须拒绝；数据面 AAD 通过 `aad_b64` 传入，服务端仅做 base64 解码、长度限制、hash 审计和一致性校验；GCM suite 额外将其作为 AEAD AAD 使用。 |
| EB-FR-06 | 系统应保证同一 GCM KeyVersion 下 AEAD nonce 不复用或满足经评审确认的低碰撞风险。 | GCM nonce 生成不可用、重复风险不可接受或阈值超限时拒绝新加密，历史解密不受影响；ECB suite 不使用 nonce。 |
| EB-FR-07 | 系统应实现基础认证授权。 | 支持 JWT/OIDC 或 HMAC 请求签名；scope、租户 ABAC、重放和跨租户负向测试通过。 |
| EB-FR-08 | 系统应实现租户信封配置。 | 支持查询和更新，使用乐观版本控制；配置项应包含默认格式、允许格式和按格式分组的 profiles。 |
| EB-FR-09 | 系统应实现审计链与日志脱敏。 | 关键事件有审计记录；审计链可验证；日志扫描不含 token、明文、DEK、CRK；授权管理调用可按需删除基线审计事件。 |
| EB-FR-10 | 系统应实现签名策略包治理。 | 未签名、错误签名、语义降级和回滚路径均有测试；加载失败保留最后有效策略；策略矩阵信息可查询。 |
| EB-FR-11 | 系统应实现异步生命周期任务。 | 任务先持久化状态，再按幂等语义执行；重试、崩溃和并发领取不得导致重复副作用；worker 运行参数可查询。 |
| EB-FR-12 | 系统应实现四平面逻辑隔离。 | 管理 API 不能执行数据面热路径；数据 API 不能改变 Key 生命周期；数据面不能直连 TPM/CRK；密钥面不做业务鉴权。 |
| EB-FR-13 | 系统应区分生产级 TPM provider 与工程 fallback。 | 生产级 provider 应基于 TSS/ESAPI 原生库；命令行 `tpm2-tools` provider 仅作为受控工程 fallback，并必须固定工具路径、白名单环境、限制临时目录权限、脱敏命令输出、覆写删除 CRK 临时文件、校验完整 CRK AAD digest。 |
| EB-FR-14 | 运维面应提供受控运维控制能力。 | 仅允许 `ops` plane 和 `ops:*` scope 调用白名单动作；覆盖只读诊断、CRK envelope 脱敏查看、AAD digest 修复、resolver 缓存刷新、lifecycle job 重试/取消、outbox 重放、审计链验证、数据库迁移状态查看和安全基线检查；不得暴露任意 SQL，不得回显 CRK/DEK/wrapped material 原文；状态变更前必须写入审计，审计失败必须中止。 |
| EB-FR-18 | 运维面高危动作应具备 break-glass 治理。 | `ops:breakglass` 仅用于受控应急动作；请求必须携带 reason、ticket_id 和幂等键；系统必须执行 dry-run/preflight、影响范围摘要、审计预写和结果审计；不允许绕过 24 小时销毁冷静期直接 purge Key material。 |
| EB-FR-19 | 运维面应提供数据库运维的安全封装。 | 支持迁移版本、连接延迟与连接池、主备角色、数据库/表/索引容量、锁与长事务、复制摘要、任务积压、领域一致性、审计链状态和 CRK envelope 一致性查询；不可观测指标必须显式标记，不得作为健康零值；只允许通过 repository/service 白名单动作变更状态；禁止执行任意 SQL、导出备份、查看记录值或密文字段原文、透传数据库原始错误以及修改审计链历史事件。 |
| EB-FR-20 | 系统应建立容量保护和安全失败方式。 | 热 Key、密钥面解封压力、审计积压和后台任务积压触发告警、限流或拒绝。 |
| EB-FR-21 | 系统应支持显式 Key ID 和 Key 有效期。 | 创建和受控导入可指定业务 Key ID 与有效期；查询结果展示有效期；到期状态显示为 `EXPIRED`；过期 Key 加密和解密均被拒绝。 |
| EB-FR-22 | 系统应支持过期 Key 延期。 | 授权管理调用可更新 `expires_at`；延期不得改变 KeyVersion、算法、策略或 wrapped material；操作必须审计。 |
| EB-FR-23 | 系统应支持 Envelope profile 配置。 | 租户配置支持 default_format、allowed_formats 和 profiles；外部格式仅保留 `json-v1` 与 `configurable-json-v1`；profile 通过 adapter、field_mappings 与 extensions 描述外部格式；JSON/profile Envelope 可携带扩展字段。 |
| EB-FR-24 | 系统应支持 outbox 大列表可用性。 | outbox 查询返回脱敏事件；前端可折叠展示大量事件，折叠不得改变接口语义或审计语义。 |

## 6. 认证授权矩阵

| 权限域 | 典型操作 | 最小 scope | 关键约束 |
| --- | --- | --- | --- |
| 密钥读取 | 查询 Key 元数据、列表 | `keys:read` 或管理 scope | 不返回 wrapped material、明文密钥或内部密钥面状态。 |
| 密钥创建/导入 | 创建 Key、受控导入外部密钥材料 | `keys:create` 或 `keys:manage` | 导入材料只进入密钥面封装流程。 |
| 密钥管理 | 更新元数据、延期、启用、禁用、轮转、取消销毁 | `keys:manage` | 必须校验租户归属、Key 状态和算法策略。 |
| 密钥销毁 | 计划销毁、到期销毁确认 | `keys:destroy` 或 `keys:manage` | 必须执行 24 小时冷静期、审计和幂等控制。 |
| 数据加密 | Encrypt、Batch Encrypt | `crypto:encrypt` | 只允许使用 `ACTIVE` KeyVersion。 |
| 数据解密 | Decrypt、Batch Decrypt | `crypto:decrypt` | 必须使用 Envelope 指定 KeyVersion。 |
| 策略管理 | 发布、灰度、回滚算法和租户策略 | `policy:manage` 或 `keys:manage` | 策略必须版本化、可审计、可回滚。 |
| 策略读取 | 查询当前签名策略包和 suite matrix | `policy:read`、`policy:manage` 或管理 scope | 不返回签名私钥或内部密钥材料。 |
| 审计查询 | 查询审计事件和验证报告 | `audit:read` 或管理 scope | 仅返回脱敏字段。 |
| 审计删除 | 删除基线审计事件 | `audit:manage` 或管理 scope | 仅用于授权清理场景，必须处理链头一致性。 |
| 租户配置管理 | 更新 Envelope 格式和 profiles | `tenant:manage` | 必须校验租户归属、格式注册状态、adapter 注册状态和 profile 映射合法性。 |
| 生命周期查询 | 查询 worker 任务、outbox 和运行参数 | `tenant:read`、`tenant:manage` 或管理 scope | 仅返回脱敏任务、事件和参数。 |
| 运维只读诊断 | 查询 CRK envelope 状态、TPM/provider 状态、resolver 缓存、迁移版本、lifecycle/outbox/audit 健康 | `ops:read` | 仅返回脱敏元数据、hash、计数和状态；不得返回密钥材料或任意 SQL 结果。 |
| 运维修复 | 修复 CRK AAD digest、刷新 resolver 缓存、重试失败 job、重放 outbox | `ops:repair` | 必须是白名单动作；必须审计先行；必须幂等；失败默认 fail-closed。 |
| 运维应急 | 执行 break-glass 修复、强制终止卡死任务、隔离异常节点或恢复受损缓存 | `ops:breakglass` | 必须携带 reason 和 ticket_id；必须记录影响范围、操作者和结果；不得直接导出或清除密钥材料。 |

## 7. Key 与 KeyVersion 状态机

| 对象 | 状态 | 允许行为 |
| --- | --- | --- |
| Key | `ACTIVE` | 可查询、更新元数据、加密、解密、轮转、禁用、计划销毁。 |
| Key | `EXPIRED` | 派生状态；可查询、更新元数据和延期；不得加密或解密。 |
| Key | `DISABLED` | 可查询、更新元数据、启用、计划销毁；不得新加密。 |
| Key | `DESTROY_PENDING` | 可查询、更新元数据、取消销毁、24 小时冷静期到期后执行销毁；不得新加密；解密按租户策略显式控制。 |
| Key | `DESTROYED` | 仅保留必要元数据和审计证据；不得启用、加密、解密、轮转或导出。 |
| KeyVersion | `ACTIVE` | 可用于新加密和解密。 |
| KeyVersion | `DECRYPT_ONLY` | 仅可用于历史密文解密，不得用于新加密。 |
| KeyVersion | `DISABLED` | 不得加密或解密，可恢复为允许状态需审计。 |
| KeyVersion | `DESTROYED` | wrapped material 不可恢复，不得加密或解密。 |

## 8. API 需求摘要

| API | 需求 |
| --- | --- |
| `POST /ui/api/v1/keys` | 创建 Key 和初始 KeyVersion；必须支持幂等键；可指定 `key_id` 和 `expires_at`。 |
| `POST /ui/api/v1/keys:import` | 受控导入外部密钥材料；导入材料只进入密钥面封装流程；可指定 `key_id` 和 `expires_at`。 |
| `GET /ui/api/v1/keys/{id}`、`GET /ui/api/v1/keys` | 返回 Key 元数据和有效期，永不返回 wrapped DEK。 |
| `PATCH /ui/api/v1/keys/{id}` | 更新名称、标签和有效期，不得修改租户、算法、策略、状态或版本。 |
| `DELETE /ui/api/v1/keys/{id}` | 计划销毁 Key，保留历史元数据和审计证据。 |
| `/enable`、`/disable`、`/rotate`、`/schedule-destroy`、`/cancel-destroy` | 管理状态转换、轮转和销毁计划。 |
| `POST /ui/api/v1/crypto/encrypt` | 小对象加密，返回 Envelope v1；可通过 `aad_b64` 携带 caller AAD bytes；未提供 `key_id` 时使用租户默认 `AES_256_ECB` Key。 |
| `POST /ui/api/v1/crypto/decrypt` | 验证 Envelope、`aad_b64`、状态、租户、权限后解密。 |
| `POST /ui/api/v1/crypto/encrypt-batch`、`POST /ui/api/v1/crypto/decrypt-batch` | 批量加解密，最多 100 条，采用 entries 输入模型并返回逐条结果。 |
| `GET/PUT /ui/api/v1/tenants/{id}/envelope-config` | 管理租户信封默认格式、允许格式和 profiles。 |
| `GET /ui/api/v1/envelope/formats` | 返回已注册格式。 |
| `POST /ui/api/v1/policies:reload` | 加载签名策略包。 |
| `GET /ui/api/v1/audit/events`、`DELETE /ui/api/v1/audit/events`、`GET /ui/api/v1/audit/chain/verify` | 查询、授权删除审计事件并验证审计链。 |
| `GET /ui/api/v1/lifecycle/jobs`、`GET /ui/api/v1/lifecycle/outbox`、`GET /ui/api/v1/lifecycle/config` | 查询生命周期任务、outbox 和 worker 运行参数。 |
| `GET /ui/api/v1/ops/*` | 运维面只读诊断、CRK envelope 脱敏状态、审计链/worker/DB 健康检查。 |
| `POST /ui/api/v1/ops/*` | 运维面白名单修复动作，必须按动作校验 `ops:repair` 或 `ops:breakglass`。 |

## 9. 非功能需求

1. 所有外部 JSON 输入必须拒绝重复字段、未知安全字段、无效 UTF-8、数值溢出和尾随数据。
2. 错误响应不得泄露租户内部命名、内部状态机细节、内部服务标识、SQL 错误或堆栈。
3. 跨租户资源不存在与权限不足应采用统一错误策略，避免枚举。
4. CRK 明文只能存在于 key-resolver 受控临界区，使用后必须尽快零化。
5. 高风险操作必须先审计后改变业务状态；审计不可用时 fail-closed。
6. 同步加解密对象上限为 64 KiB；批量 API 单次最多 100 条。
7. RTO/RPO 只有在生产级恢复演练后才能作为对外承诺。
8. JSON/profile Envelope 扩展字段不得参与替代核心 Envelope 字段；解密必须使用调用方传入的同一份 caller AAD bytes。外部 Envelope 格式仅保留 JSON 与 configurable JSON；服务端内部 canonical 编码只作为认证边界和解析实现细节，不作为对外 API 格式。
9. 前端页面不得作为安全边界；所有平面、scope 和租户隔离必须由后端强制执行。

## 10. 发布门禁

工程基线发布必须满足：

1. Golden Vectors、状态机、认证/重放/跨租户、nonce 异常、日志扫描和 E2E 均通过。
2. Key 创建、查询、列表、更新、计划销毁、取消销毁、启停和轮转均有自动化测试或等效验证。
3. Key 显式 ID、有效期展示、过期拒绝加解密、过期扫描和授权延期均有自动化测试或等效验证。
4. 批量 entries、部分失败、稳定错误码、Envelope 字段配置、审计链篡改检测、授权审计删除、策略 reload、worker 幂等和容量保护验证通过。
5. 数据面无 TPM/CRK 权限；四平面逻辑隔离可验证；data token 访问管理 API、普通 token 修改租户配置等负向用例必须失败。
6. 明文密钥导出请求、完整 Envelope 日志记录、wrapped material 回显等路径均被拒绝。

## 11. 需求变更准入规则

任何新增或变更需求必须写明：

1. 工程基线影响范围。
2. API 或数据模型。
3. 影响的隔离面。
4. 安全不变量。
5. 测试证据。
6. 指标与告警。
7. 回滚动作。

缺少任一项不得进入基线验收。
